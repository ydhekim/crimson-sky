Acceptance criteria: `MatchmakingRequest`/`MatchmakingFoundResponse` round-trip; pairing prefers closest Elo available, falls back to widening range after a timeout (define timeout value when implementing).

## Epic C — Persistence & Rewards — M3

**C1.** As a player, my Gold, Exp, and Elo update after a battle concludes.
Acceptance criteria: `battle_history` row inserted; `accounts.global_currency` and `characters.experience`/`characters.elo` updated in the same transaction (see system design §8 for the account/character split, §8.1 for the exact Elo/Gold/Exp formulas — decided 2026-07-10: standard Elo rating formula K=32, flat-plus-Elo-bonus Gold/Exp on a win, small flat consolation Gold/Exp on a loss). Scope is deliberately raw-number accumulation only — no leveling mechanic (level-up thresholds/consequences) is part of this story; that's a separate future epic.
Priority: P0. Size: M. Status: done. *(New `RewardService` wraps `AttackService` without touching it: `AttackRequestHandler` resolves the battle, then calls `applyRewards(AttackResult)` and folds the returned `RewardOutcome(goldDelta, expDelta, eloDelta)` into the response. The §8.1 formulas are implemented verbatim (Elo K=32; win = 25 gold / 50 exp plus a floored Elo-gap bonus; loss = flat 5/10 with the standard formula producing the negative Elo itself), with a bot's Elo defined as the attacker's own. **`RewardService` takes the raw `Jdbi`, not the other services** — `CharacterService`/`AccountService` wrap `onDemand()` proxies that open a connection per call, so using them in sequence could leave a half-applied reward behind; all three writes (`CharacterDao.addExperienceAndElo`, `AccountDao.addGlobalCurrency`, new `BattleHistoryDao.insert`) go through DAOs attached to one `useTransaction` handle. Both updates are atomic SQL increments, not read-then-writes. Only the attacker's side is ever written: an async attack is one-sided, so the opponent (real or bot) is never credited, debited, or given a row. A failed reward transaction logs loudly and pays a zero `RewardOutcome` rather than propagating — the player still gets the battle they just fought.)*

**C1 wire-format change, worth calling out:** `AttackResponse` gained `goldDelta`/`expDelta`/`eloDelta` — the packet previously carried no rewards at all, so the client had nothing to show. `RecordSerializer` picks the new components up with no `KryoConfig` change. The §7 transparency guarantee survives: a bot fight's Elo delta is computed against the attacker's own rating, which is exactly what an evenly matched real fight produces, so the deltas leak nothing. This did force a change to two B4 tests (the `toResponse()` signature and `AttackServiceTest`'s reflective wire-field assertion, which is *supposed* to fail when the packet's shape changes) — everything else in B4 passes untouched.

**C2.** As a player, items "lost" in battle via skills are not actually removed from my permanent inventory.
Acceptance criteria: explicit test that a battle involving item loss leaves `characters.inventory` unchanged (GDD §4 rule).
Priority: P1. Size: S. Status: done. *(Reframed, deliberately: no skill or mechanic in the game causes item loss yet — Break/Steal are deferred to Epic J — so there is no item-loss scenario to fight through end to end, and inventing one would test a mechanic that doesn't exist instead of the rule that does. `BattleLeavesInventoryAloneTest` locks in the property C2 actually protects and that *is* testable today: a full `AttackService.attack()` + `RewardService.applyRewards()` round trip leaves the stored `inventory` and `loadout` JSON byte-for-byte identical, asserted alongside proof that the same round trip really did write `experience`/`elo`/`global_currency`/`battle_history` (otherwise the assertion would pass vacuously on a flow that wrote nothing). A second, structural assertion reflects over `CharacterDao` and fails if any `UPDATE` statement so much as mentions `inventory`/`loadout` — so a future item-loss skill can't start writing items by accident; it has to change this test on purpose, which is the conversation C2 exists to force.)*

**C3.** As a developer, `battle_history` exists so future stats/leaderboard features have data to read.
Acceptance criteria: migration V8 (`V8__Battle_History.sql`, per system design §8 — V6/V7 already went to Stamina/Elo) applied cleanly on a fresh DB and on top of V1–V7.
Priority: P2. Size: S. Status: done. *(`V8__Battle_History.sql` added with §8's schema verbatim — `SERIAL`/`INTEGER` keys per `V1`'s actual conventions, nullable `opponent_character_id` (a bot has no row in `characters`), server-side-only `opponent_is_bot`, and the three deltas mirrored so a payout can be audited without reconstructing it from two running totals. `accounts.global_currency` already existed in V1; no migration needed for the gold wallet.)*

*Close-out flag on the numbers, per the C1 prompt's own instruction not to silently re-tune them: with real values flowing through the tests, an evenly matched win pays 25 gold / 50 exp / +16 Elo and a loss pays 5 / 10 / −16. Nothing looked broken, but two things are worth a deliberate look during the tuning pass rather than a quiet adjustment now. (1) **K=32 is fast for an attack-on-demand game** — it's calibrated for chess, where a player might rate 30 games a season; here a player can fight several times a minute, so ratings will swing hard and the ±100 Elo opponent band will churn quickly. Consider K=16, or a K that decays as a character's battle count grows, once there's play data. (2) **The Elo-gap bonus is generous at the extremes**: beating an opponent 500 Elo above you pays 75 gold / 150 exp, 3x the base, and nothing caps the gap after the unbounded widening step (B4) matches you against someone arbitrarily far away when the band is empty — on a thin population, that's the shape of a farm. Both are first-pass constants by design (§8.1) and neither is worth changing before there's a Gold sink and a level curve to validate against.*

## Epic N — Loadout Constraints: Weight Capacity & Weapon Durability — new, third slice of the 2026-07-10 a–k scope expansion

Two new axes layered on the existing weapon pouch, designed together since both modify the same layer. See system design §17 for the full formulas and reasoning. Neither replaces §4.3's soft per-item weight penalty or Stamina's in-battle rotation — both stand unchanged.

**N1.** As a player, my total carry weight across equipped weapons is capped by my Strength, and an over-budget weapon can't be added to my loadout at all.
Acceptance criteria: `maxCarryWeight = strength × 3` plus any equipped `WEIGHT_CAPACITY_BONUS` passive bonus (system design §17); enforced at loadout-save time as a hard rejection (distinct from and layered on top of §4.3's existing soft `comfortableWeight` per-item penalty, which still applies independently). Weapons only — skills/pets have no `weight` dimension.
Priority: P1. Size: S. Status: **done** (2026-07-16). `CharacterService.saveLoadout` gained the gate between the existing skill-slot-count check and the item-ownership check: sum of `loadout.weapons()`'s `weight()` vs `strength × 3 + PassiveEffects.totalWeightCapacityBonus(loadout)`, rejected with the new `MessageCode.LOADOUT_WEIGHT_EXCEEDED`. The cap is inclusive (`>`, not `>=`). §4.3's soft per-item `comfortableWeight` penalty in `CombatMath.effectiveStrength` is untouched, as designed — the two coexist.

**Unplanned extraction this story forced, worth recording: `core.combat.PassiveEffects`.** The passive-bonus term is the same sum `BattleParticipant.fromCharacter` was already computing inline (M3), but N1 needs it at *loadout-save time* — a plain server-side call with no Ashley `Engine`, so it can't go through `BattleParticipant`. The choice was to duplicate the aggregation or extract it; duplicating it would have meant two independent implementations of the number that decides what a player may equip and the number combat then plays with, free to drift apart silently on the next edit. So the aggregation moved into a new pure, Ashley-free class (`totalWeightCapacityBonus`/`totalDodgeChanceBonus`/`totalCritChanceBonus`/`totalResourceCostReduction`/`applyStatBonuses`), in the free-standing style of `CombatMath`/`DamageCalculator`, reachable from `server` (which already depends on `core`). `fromCharacter` now calls it instead of its inline loop — behavior-preserving, and the M-pass `BattleParticipantPassivesTest` was re-run unmodified as the regression check, alongside a new `PassiveEffectsTest`. One formula, one place, both callers agree by construction.

**N2.** As a player, my weapons wear down with use and stop working until I repair them.
Acceptance criteria: `Weapon` gains `maxDurability`/`currentDurability` (flat `maxDurability = 20` for all starter weapons, first pass); any weapon that fired at least once in a battle loses 1 durability, applied once per battle regardless of hit count. At 0 durability, the weapon is treated as unaffordable in the pouch cascade — same mechanism as insufficient Stamina, no new branch. Repair (shop action, price deferred to the shop epic) resets `currentDurability` to `maxDurability`.

Requires two things not yet in the codebase: (1) `ResolvedAction` needs a new item-id field — today it only carries a coarse `ActionSource` category and a display label, nothing identifies *which* weapon fired when a pouch holds several; (2) `CharacterDao.updateInventory(long characterId, String inventoryJson)` — a JSONB read-modify-write (not an atomic increment like gold/exp/Elo), locking the character row (`SELECT ... FOR UPDATE`) inside `RewardService`'s existing transaction rather than building real optimistic-concurrency handling (safe given the daily battle cap makes concurrent attacks on one character unrealistic). This is the *second* capability needing a deliberate, named exception in `BattleLeavesInventoryAloneTest` (C2), after L3's item-grant.

Priority: P1. Size: M. Status: **done** (2026-07-16). `Weapon` gained `maxDurability`/`currentDurability` (flat 20 for all content, per §17's first pass); `CombatMath.isAffordable(Weapon, int)` gained `weapon.currentDurability() > 0`, so a broken weapon is skipped by the exact code that already skips an unaffordable one — no new branch, no RNG-consumption change, and no existing assertion moved. `ResolvedAction` gained `long itemId`, populated for `WEAPON`/`SKILL`/`PET` and `0L` for `PUNCH`/Burned; populating it uniformly (not weapons-only) costs nothing now and hands Epic O's consumable-charge tracking the same capability for free. `RewardService` scans `result.turns()` for distinct `WEAPON`-source ids (a `Set` — a weapon that fired on all 40 turns still wears exactly 1, per §17) and decrements inside its existing transaction.

**Two prerequisites §17 predicted turned out not to be needed, and the reasons are worth keeping.** (1) **No new `CharacterDao` method and no second C2 exception.** §17 anticipated `updateInventory` as new work and "another deliberate, named exception" in `BattleLeavesInventoryAloneTest`; L3 had already built the method, and §18 had already settled the rule that supersedes §17 here — every feature that mutates `inventory` from now on is *a different in-memory transformation before the same write call*, not a new write path. So the sanctioned-writer list is still exactly two (`updateInventory`, `updateLoadout`) and `onlyTheSanctionedGrantPathCanReachTheStoredItems` passes unmodified. The milestone grant and the durability decrement now share one locked read and one write, wearing before granting so a weapon handed out *as* this battle's bonus isn't charged for a battle it wasn't in. (2) **No `AttackResponse` "your weapon broke" field** this pass — same treatment as the M-pass crit flag, deferred to the not-yet-built combat screen (M4).

**Design ambiguity resolved during implementation — `Inventory`, not `Loadout`, is the single source of truth for durability** (system design §17 updated with this). §17 specified durability as a `Weapon` field but never said where the mutable state lives when `Loadout` and `Inventory` each hold their own independent copy of the same weapon by id. It matters: `fromCharacter` read combat weapons straight from `Loadout`, and `saveLoadout` (M3) lets a client submit its own `Weapon` object into `Loadout` at any time — so a post-battle decrement written only to `Inventory` would never be seen by combat, which would keep reading `Loadout`'s pristine copy. A client could then hold a permanently unbreakable weapon by re-saving its loadout. Resolution: `Loadout` means only "this item id is equipped"; `fromCharacter` cross-references each equipped weapon's id against `character.inventory()` and takes *that* copy's `currentDurability` (falling back to the `Loadout` copy only when the id isn't in inventory — defensive for real characters since `saveLoadout` enforces item ownership, but the case every `BotFactory` bot legitimately hits, since a bot has a loadout and an empty inventory and must never start a fight holding a broken weapon). All durability reads and writes therefore go through `Inventory` alone — no dual-write, nothing to keep in sync.

*Adjacent hole this surfaced but deliberately did not fix (candidate for Epic I/K):* the same cross-reference makes it plain that `saveLoadout` validates a submitted item by **id only**, so a client can submit a `Loadout` weapon with a real id but inflated `minAttack`/`maxAttack`/`weight`, and combat will use those numbers. Durability is now immune (it's re-read from inventory), but nothing else is. The fix is for `fromCharacter` to take the whole weapon from inventory rather than just its durability, or for `saveLoadout` to store ids instead of records — both are real changes beyond N1/N2's scope, and neither is urgent while the transport is still plaintext and unauthenticated at the network layer (Epic I).

*(Test note: every `Weapon` fixture across `core`/`server` was updated to `20, 20` rather than allowed to default — a fixture silently shipping at `currentDurability = 0` would read as broken and quietly drop its scenario to Punch, making the assertion pass for the wrong reason. `BattleLeavesInventoryAloneTest`'s round-trip test was narrowed on purpose: it asserted the stored inventory JSON was byte-for-byte identical after a battle, which durability makes **wrong** rather than merely violated — a battle now legitimately writes one field of one item. It now asserts what C2 was always defending (no stored item is ever taken away, and the item set is unchanged), keeps the byte-for-byte check on `loadout` where a battle still has no business writing, and gains a positive test pinning the 20 → 19 decrement in the persisted JSON.)*

## Epic O — Shop, Potions & Pet Health — new, fourth slice of the 2026-07-10 a–k scope expansion

The shop needed real prices before it could be built — this epic exists because §16 (skill points) and §17 (durability) finally gave it some. Potions and pet health are bundled in because both turned out to hinge on the same question (what "restore a resource" means when a battle resolves in one round trip). See system design §18 for the full reasoning.

> **Groundwork complete as of 2026-07-17 (O1/O2/O3 done).** Everything this epic designed now exists in code except **O4's earning side, which is Epic P's (quests) to build** — O4's redemption half shipped with O1/O3, so when quests land they need to add a grant and nothing more. Splitting O2 out of the O1/O3 pass paid off exactly as predicted: O1/O3 were close mirrors of shipped patterns (L1–L3's guarded-spend, N2's wear shape) and carried little structural risk, while O2 really did need the structurally novel pieces — a self-targeting heal in a `BattleEngine` shaped entirely around attacker-deals-damage-to-defender, a new pre-cascade conditional check, a new `ActionSource`, and a second meaning for `ResolvedAction.damage`. Worth noting what the epic as a whole leaves behind: **repair, restocking and acquisition are three different questions**, and only repair got answered. A spent potion has no repair by design (§18) and no shop entry — it is replaced, not refilled — but nothing sells one yet either, so the whole potion mechanic sits complete-but-unreachable until content authoring exists (Epic E). Same for pets: nothing grants one outside character creation.

**O1.** As a player, I can spend gold in a shop to repair weapons, buy skill-restoration scrolls, and reset my skill tree.
Acceptance criteria: repair costs `5 gold × missingDurability` (targeted at a specific owned weapon, not a catalog purchase); skill-restoration scroll costs 50 gold to buy (separate from M4's usage fee); skill tree reset token costs 1000 gold, consumed to trigger M5. Shop is gold-only — no premium currency sold for anything in it (keeps IAP scoped to the account-level levers already decided separately, never to combat power). Repair must accept a Repair Token as an alternate payment to gold (see O4) from day one, not bolted on later.
Priority: P1. Size: M. Status: **done** (2026-07-16). New `ShopService` (raw `Jdbi`, same justification as `RewardService`/`SkillTreeService`: a repair or purchase spans `characters.inventory` and `accounts.global_currency`, and `onDemand` DAO proxies take a connection per call, so only DAOs attached to one `useTransaction` handle commit together). Four operations — `repairWeapon`/`repairPet` (both `useToken`-parameterized) and `buyScroll`/`buyResetToken` — behind four new packet pairs and four handlers following `LearnSkillNodeRequestHandler`'s exact shape (unauthenticated connection and non-owned character are logged-and-dropped, never answered; every actionable refusal is answered so the client can explain it). Prices are §18's verbatim: `5 gold × missing`, 50, 1000. Gold-only throughout — no premium currency touches any of it.

**The token-redemption path is real and callable today, and that was the point of building it now.** Nothing yet grants a Repair Token or a Pet Care Kit — that's O4/Epic P — so `useToken = true` can only currently fail with `SHOP_TOKEN_INSUFFICIENT` in a live game. It is still fully implemented and tested (redeeming decrements the count, spends no gold, and a repair with 0 tokens is rejected rather than silently falling back to gold). §18 and O1's own acceptance criteria both call for this explicitly: the alternate payment path is a shape the repair action has from day one, not an argument threaded through later. When O4 lands it needs to add a grant, nothing more.

**Consumable counts live on `Inventory`, not a new table, and no migration was needed anywhere in this pass.** `Inventory` gained `Map<String, Integer> consumables` (key → count, the same shape as `Character.skillTree`, which already proves it round-trips through both Jackson `@Json` and Kryo with no extra registration). Keys are four `public static final String` constants on `ShopService` rather than an enum — four ad-hoc keys in a map nothing switches over. This is exactly the embedded-JSONB forward compatibility §18 banked on: everything here is a `Weapon`/`Pet`/`Inventory` shape change plus the `getInventoryForUpdate`/`updateInventory` primitive N2 already established. No new `CharacterDao` method, no third C2 exception.

**The load-bearing implementation decision: the row lock is the only guard a JSONB field gets, so re-verify against the locked read, not the pre-check.** `AccountDao.spendGlobalCurrency`'s `WHERE global_currency >= :cost` is a real SQL guard against a concurrent gold race — a consumable count buried in a JSONB blob has no equivalent, and the only thing serializing two concurrent repairs of one character is `getInventoryForUpdate`'s `FOR UPDATE`. So the friendly "can this even succeed" checks (ownership, item exists, cost, balance) run against a plain non-locked read for a clean rejection reason — `SkillTreeService`'s style exactly — but every decrement and repair re-reads the token count *and* the target item from the locked copy inside the transaction and acts on that, never on the pre-check's. A disagreement means someone else's repair landed first and throws `IllegalStateException`, caught by the outer handler and reported as `ERROR_UNKNOWN` — the same treatment a lost gold race already gets in `SkillTreeService.learnOrUpgrade`. Getting this backwards would be a silent double-spend of a token, and it would never show up in a single-threaded test.

*(Also: a no-op repair is rejected with `SHOP_NOTHING_TO_REPAIR` rather than charging 0 gold or, worse, burning a token for nothing. Every write here goes through `updateInventory`, so the C2 sanctioned-writer list is still exactly two.)*

**O2.** As a player, I can equip potions that automatically heal me mid-battle when a resource drops below a threshold I set.
Acceptance criteria: new `SkillType.CONSUMABLE` alongside `ACTIVE`/`PASSIVE` (system design §18) — `Skill` gains `restoresResource` (`HEALTH`/`MANA`/`STAMINA`), `thresholdPercent`, `restoreAmount` (a flat quantity — no roll, no stat bonus, sized by content: e.g. Small/Medium/Large Health Potion at 100/200/300), `charges`. Before the existing weapon/skill cascade each turn, equipped `CONSUMABLE` skills are checked in priority order; the first one below its threshold with `charges > 0` fires, heals, consumes a charge, and becomes that turn's entire action (existing cascade skipped that turn; pet action unaffected). No trigger, no change to existing behavior. Shares M3's 5-slot pool with actives/passives. Unlike weapon durability/pet health, `charges` depletes once per actual trigger (not capped at one per battle) and is persisted — `Inventory` is the source of truth, same drift-resolution pattern as N2/O3.
Priority: P1. Size: M. Status: **done** (2026-07-17). Built exactly as scoped: `SkillType.CONSUMABLE` and `ActionSource.CONSUMABLE` appended (never inserted — Kryo registers enums positionally, §5), `Skill` gained the four trailing fields plus `consumed(int)`/`withCharges(int)`, and the reactive logic stayed isolated to one pre-cascade check in `BattleEngine.resolveResultSet` delegating to a new pure `core.combat.ConsumableResolver` (Ashley-free, sibling of `ActionResolver`/`PetResolver`), with a new `ConsumableSlotComponent` pouch beside `SkillSlotComponent`. Three things worth carrying forward:

- **The restore amount is flat, and that is why a potion is the only branch of the cascade that draws no RNG.** §18's original sketch reused damage's `min`/`max`-plus-stat-bonus shape; the 2026-07-16 amendment superseded it. Potency belongs to the potion, not the drinker, so Small/Medium/Large are three authored instances rather than a formula — the same content-authoring pattern the three starter weapons already use. `ConsumableResolver.chooseConsumable` therefore takes no `SplittableRandom` at all, which `ConsumableResolverTest` pins **structurally** (by reflecting over the signatures) as well as behaviorally: a build carrying a potion that never fires must resolve a battle byte-for-byte identically to a build carrying none, and a single stray draw in this path would desync every roll after it.
- **`charges` is a tally, not a set — the one place potions genuinely diverge from N2/O3.** Weapon durability and pet health lose at most one point per *battle* however often the item acted (hence a `Set` of fired ids); a potion's charges are spent per *trigger*, because a long fight can genuinely cross the threshold twice. So `RewardService` gained `consumableTriggerCounts(result)` (a `Map<Long,Integer>`, `merge(id, 1, Integer::sum)`) next to `firedWeaponIds`/`firedPetId`, and a third loop inside the *same* locked read and single `updateInventory` call the other two already share. `Inventory` stays the source of truth (`BattleParticipant.atInventoryCharges`, the third sibling of `atInventoryDurability`/`atInventoryPetHealth`); `ConsumableSlotComponent.remainingCharges` is the battle-scoped count seeded from it, which is what stops a 2-charge potion healing a third time in one fight. C2 needed **no** new exception — still exactly two sanctioned writers, per §18's standing rule.
- **Implementation note the prompt's own sketch got subtly wrong:** applying the character's entry *inside* the potion branch (as sketched) would have moved the character's dodge/damage draws ahead of the pet's gate roll, changing RNG consumption order for every existing potion-less battle. The built version decides first (potion check → cascade), then keeps Step 2's pet decision and Step 3's apply exactly where they were, so the no-potion path is byte-for-byte the pre-O2 cascade — which `DeterminismTest`/`BattleEngineTest`/`ActionResolverTest`'s untouched fixtures confirm.

**Potion content is still nobody's job.** Nothing authors a Small/Medium/Large Health Potion — or any potion at all — and nothing grants one: the mechanic is complete and tested, but no player can obtain a potion in a live game today. That is the same "data-driven content is Epic E's problem" deferral every other item type in this codebase already carries (`BotFactory`'s three hardcoded weapons and skills, L3's duplicated bonus-weapon table), not an oversight to fix here. Acquisition is a content-authoring question; when Epic E lands, a potion is one more row in the same table.

**O3.** As a player, my pets tire from use and need restoring, the same way weapons do.
Acceptance criteria: `Pet.healthPoint` becomes real (max health); new `currentHealth` depletes by 1 per battle the pet acts in (not per hit). At 0, the pet's bonus action is skipped that battle — soft, never blocks attacking. Repair via O1's shop action (`5 gold × missingPetHealth`, or Pet Care Kit redemption per O4). Explicitly out of scope: `Pet.defence` stays unresolved — this is wear from use, not damage taken from an opponent; a real pet-vs-opponent targeting mechanic is a separate future question.
Priority: P2. Size: S. Status: **done** (2026-07-16, with O1). Deliberately built as durability's mirror at every level, because §18 designed it as one: `Pet` gained `currentHealth` alongside `worn()`/`withCurrentHealth()`/`repaired()` (and `Weapon` gained the matching `repaired()`); `CombatMath.isPetUsable(Pet)` sits beside `isAffordable(Weapon, int)`; `PetResolver.choosePetAction` gained a guard right after its existing `pet == null` check, so a worn-out pet is skipped **with no gate roll consumed** — the same treatment as no pet at all, which is what makes §18's "the bonus action is simply skipped" true without perturbing the seeded RNG stream a healthy build would draw from. `RewardService` gained `firedPetId(result)` next to `firedWeaponIds(result)` (a `Set`, so a pet acting on all 40 turns still wears exactly 1) and a pets loop inside the *same* locked read and single `updateInventory` call the weapon decrement and item grant already share.

**`Inventory` is the single source of truth for pet health too** — the identical cross-reference N2 resolved for durability, for identical reasons: `Loadout` and `Inventory` each hold their own copy of a pet, `saveLoadout` lets a client submit its `Loadout` copy at any time, and the post-battle decrement only ever writes `Inventory`. `BattleParticipant.fromCharacter` gained `atInventoryPetHealth` beside `atInventoryDurability`, and `PetSlotComponent.currentHealth` is now *seeded from* the persisted value instead of always-full `healthPoint()`. What did **not** change: nothing in combat decrements that battle-scoped copy — §18's "a real pets-can-be-hurt-by-the-opponent mechanic is a separate question" stands, and `Pet.defence` stays exactly as vestigial as it was. This depletion is wear from use, not damage taken.

*(Test note, same caution N2's pass called out for weapons: every `Pet` fixture was updated to ship at full health rather than allowed to default — one silently at `currentHealth = 0` would read as worn out and quietly stop acting, making every pet-resolution test pass for the wrong reason.)*

**O4.** As a player, weekly quests can reward tokens that let me repair a weapon or restore a pet for free.
Acceptance criteria: a Repair Token (redeems for a full weapon repair) and a Pet Care Kit (full pet health restore), usable as alternate payment in O1/O3's repair actions instead of gold. Weekly cadence against the ~5-day depletion cycle (N2) gives a real free relief valve without eliminating the reason to ever pay — deliberately not a daily/automatic regen, which would trivialize the sink entirely (considered and rejected during design). Exact weekly reward table and how tokens are earned is the quest epic's job, not yet designed — this story only needs the redemption mechanic itself to exist in O1/O3.
Priority: P2. Size: S. Status: todo. *(No longer depends on O1/O3 — the redemption half shipped with them on 2026-07-16: `repairWeapon`/`repairPet` already accept a token in place of gold, decrement the right `consumables` key, and reject cleanly at 0. What's left here is purely the earning side, which depends on the not-yet-built quest epic (P). As of 2026-07-17 this is the **only** part of Epic O still outstanding — O1/O2/O3 are all done.)*

## Epic P — Quest System — new, fifth slice of the 2026-07-10 a–k scope expansion

Three quests total for v1.0, deliberately small. See system design §19 — progress is computed live from `battle_history`, no new tracking needed; only claiming needs new state.

**P1.** As a developer, quest progress is computed from existing battle data, not duplicated tracking.

Priority: P0. Size: M. Status: **done** (2026-07-19). *(`QuestPeriods` (`server.quest`) owns the shared UTC-midnight / most-recent-Monday boundary math, with package-private `Clock`-taking overloads as the test seam — Epic Q's daily battle cap imports `startOfToday()` verbatim. `V11` adds `quest_claims` and `QuestClaimDao` (`isClaimed` for the daily/weekly guard, `countClaimsSince` for the repeatable cap). **A real gap found and fixed while grounding P1 (system design §0 / §19).** §19's own live-progress query assumes a `battle_history.won` column that **did not exist** — `V8` never recorded win/loss, and `RewardService.applyRewards`'s insert never wrote one. Fixed properly: `V11` adds `won BOOLEAN NOT NULL DEFAULT FALSE`, `BattleHistoryDao.insert` gained a `won` parameter, and `RewardService` now passes `result.won()` through (a pure wiring gap — the value was already sitting on `AttackResult`). `DEFAULT FALSE` means pre-migration rows read as losses for quest-counting — harmless pre-alpha (no real player history) and self-correcting as new rows carry the real value. `BattleHistoryDaoCountWinsTest` is the test that would have caught this had it existed before.

**P2.** As a player, I can see my current quest progress and claim completed quests for their rewards.

Priority: P0. Size: M. Status: **done** (2026-07-19). *(Depends on P1.)* *(`QuestService` (`server.service`, raw `Jdbi` for the same cross-table-atomicity reason as `ShopService`/`RewardService`) with `getStatus`/`claim`; packets `QuestStatusRequest`/`Response` + `ClaimQuestRequest`/`Response`, handlers `QuestStatusRequestHandler`/`ClaimQuestRequestHandler` following `RepairWeaponRequestHandler`'s drop-or-answer posture; five new `MessageCode`s. **P3.** As a player, three quests are available: a daily, a weekly, and a repeatable.

Priority: P0. Size: S. Status: **done** (2026-07-19). *(`QuestCatalog`/`QuestDefinition`/`QuestPeriodType` (`server.quest`) curate the three quests in code — same treatment as `BotFactory`'s starter content and `SkillTreeCatalog`'s nodes, Epic E makes it data-driven later.)*

**Reward decisions made this pass (real gaps §19 left open, resolved the same way L3's bonus table resolved an equivalent one — real content only):**
- **Daily** → 1 skill-restoration scroll (`Inventory.consumables["skill_restoration_scroll"]` +1). `QuestService` sits in `server.service` specifically to reuse `ShopService`'s package-private consumable-key constants rather than duplicating the strings.
- **Weekly** → the player's choice of **exactly two** rewards, a Repair Token **or** a Pet Care Kit (`ShopService.REPAIR_TOKEN`/`PET_CARE_KIT`). **No weapon/pet option** — deliberately kept to the two existing maintenance-economy consumables rather than reaching into `RewardService`'s/`BotFactory`'s weapon/pet constants, which are calibrated for starter content / milestone bonuses and were never meant to double as quest rewards. An invalid/absent `rewardChoice` is rejected (`QUEST_INVALID_REWARD_CHOICE`).
- **Repeatable** → a flat **15 gold** via `AccountDao.addGlobalCurrency`, **capped at 3 claims/day**. §19 also names "a potion charge" as an alternative — **not implemented**: O2 built the potion *mechanic* but no potion `Skill` instance has ever been authored (nothing analogous to the real weapons/pets other grants use exists yet), so there is nothing concrete to grant. Gold-only for this pass — flagged the same way L3 flagged pets/scrolls as "no acquisition mechanic in code yet."

**Note on the repeatable quest's cap — different mechanism than daily/weekly, not the same `UNIQUE` trick.** Daily/weekly rely on `quest_claims`' `UNIQUE (character_id, quest_id, period_start)` to allow exactly one claim per period — that only works because `period_start` is the same value for every claim attempt within the period, so a second insert collides and fails. A repeatable quest needs the opposite: multiple rows for the same `quest_id` on the same day. Each repeatable claim takes its own `period_start` (`Instant.now()`, its own `claimed_at` moment, not a shared day-boundary value) so the rows don't collide, and the cap is an application-level check — `QuestClaimDao.countClaimsSince(characterId, questId, startOfToday)`, reject once it's already 3. *(Built exactly as described; `QuestServiceTest` asserts two same-day repeatable claims land two distinct `period_start`s under the one `quest_id`.)*

**Natural next slice: Epic Q (daily battle cap, character slots).** It only needs to import `QuestPeriods.startOfToday()` (already built and shared here) and add the two bonus columns (`characters.bonus_daily_battles`, `accounts.bonus_character_slots`) per system design §20 — nothing in Epic P blocks it. Note also that none of Epic P's three quests grant `bonus_character_slots`/`bonus_daily_battles` today (system design §20 lists quest rewards as an available extension point, left unchanged).

## Epic Q — Account Levers: Daily Battle Cap & Character Slots — new, sixth slice of the 2026-07-10 a–k scope expansion

Smaller than the preceding epics — the counting mechanism was already settled by Epic P (reuse `battle_history`'s period-boundary query). What's new here is extensibility itself. See system design §20.

**Q1.** As a player, I can only fight up to a daily battle limit per character, and that limit can be permanently increased.
Acceptance criteria: base cap 5, extended by `characters.bonus_daily_battles INTEGER NOT NULL DEFAULT 0` (effective cap = `5 + bonus_daily_battles`). `AttackService` checks `COUNT(*) FROM battle_history WHERE character_id = ? AND created_at > startOfToday` (unfiltered by `won`) against the effective cap before resolving the battle — rejects early with a distinct reason code (not a silent drop), exact packet shape deferred to implementation-prompt time. Migration `V12__Add_Account_Progression_Levers.sql` adds this column.
Priority: P0. Size: S. Status: **done** (2026-07-20). *(Depends on Epic P's period-boundary query pattern.)*

**Q2.** As a player, my account's character slot count can be permanently increased beyond the starting 3.
Acceptance criteria: no new column — reuses the existing `accounts.max_slots INTEGER DEFAULT 3` (V1 migration), which was already read into `Account.maxSlots()` and sent to the client but never actually enforced (`CharacterService.createCharacter` had its own independent hardcoded `>= 3`, a real display/enforcement gap found while grounding this epic — see system design §20's 2026-07-20 correction). `CharacterService.createCharacter`'s check becomes `getCharacterCount(accountId) >= maxSlots`, where `maxSlots` is `connection.account.maxSlots()`; granting a bonus slot means incrementing `max_slots` directly (`AccountDao.addCharacterSlots`).
Priority: P0. Size: S. Status: **done** (2026-07-20).

**Q3.** As a player, IAP purchases, achievement unlocks, and quest rewards can each grant these bonuses.
Acceptance criteria: Q1's `bonus_daily_battles` column and Q2's `max_slots` column are the shared targets every grant source writes into, via new atomic-increment DAO methods (`CharacterDao.addBonusDailyBattles`, `AccountDao.addCharacterSlots`) — this story only needs those write paths to exist, not every source wired up. IAP fulfillment is a platform concern (Epic G/Steam), out of scope here. Achievement-triggered grants depend on unlock-detection/reward-granting logic that does not exist yet — confirmed by reading `AchievementService`, which today only exposes `getPlayerAchievements` with no reward path — real dependency on the not-yet-designed achievements epic (item j). Quest-triggered grants can reuse the existing L3/O4 reward-application primitives; none of Epic P's three locked quests currently grant either bonus, and this story does not change Epic P's content.
Priority: P2. Size: S. Status: **done** (2026-07-20) — the in-scope deliverable (the atomic-increment write paths) exists; no grant source is wired yet, as scoped. *(Achievement-grant path still blocked on the achievements epic; IAP and quest paths can wire in independently now that Q1/Q2 exist.)*

## Epic R — Ranked Ladder — new, seventh slice of the 2026-07-10 a–k scope expansion

A second, opt-in Elo track layered on top of the existing one, unlocked at level 25. See system design §21 for the full reasoning, including why this is additive (normal battles and `characters.elo` are untouched) rather than a retrofit.

**R1.** As a player who has reached level 25, I can choose to fight a ranked battle instead of a normal one.
Acceptance criteria: `AttackRequest` gains `BattleMode mode` (`NORMAL` default, `RANKED`); server rejects `RANKED` when `character.level < 25`, same guardrail style as the existing ownership check. Per-battle Gold/Exp/skill-point rewards are identical regardless of mode — no new reward-tuning knob.
Priority: P1. Size: M. Status: done (2026-07-20). *(Depends on Epic L for `character.level` to exist. Rejection reuses Q1's `AttackRejectedResponse` with new reason code `RANKED_LEVEL_GATE_NOT_MET`.)*

**R2.** As a player, my ranked battles are matched against other ranked-eligible opponents by ranked Elo, separate from my normal matchmaking Elo.
Acceptance criteria: `battle_history` gains `battle_mode VARCHAR(16) NOT NULL DEFAULT 'NORMAL'` and nullable `ranked_elo_delta INTEGER` (migration `V13__Add_Ranked_Battle_Mode_And_Ladder_Claims.sql`, set only on `RANKED` rows). Ranked Elo is never a mutable column — it's computed live as `1000 + SUM(ranked_elo_delta) WHERE character_id = ? AND battle_mode = 'RANKED'`, same "compute from `battle_history`, no duplicate tracking state" pattern already used for quests (Epic P) and the daily battle cap (Epic Q). Ranked opponent selection reuses `AttackService`'s existing band→widen→bot algorithm, keyed off ranked Elo and restricted to level-25+ candidates (new DAO query, same shape as `findOpponentCandidates`); bot fallback uses `BotFactory.createBot(rankedElo)`, no `BotFactory` code change needed.
Priority: P1. Size: M. Status: done (2026-07-20). *(Depends on R1. Reuses AttackService's existing matching algorithm and BotFactory as-is. Shipped as `V13__Add_Ranked_Battle_Mode.sql` — the `ladder_claims` half of the originally sketched migration name belongs to R3, which gets its own V14+.)*

**R3.** As a player, I can see a monthly ladder ranking ranked-eligible characters by ranked Elo, and claim a reward once per month based on my standing.
Acceptance criteria: ladder standing (current or as of a past month's end) is computed via the same bounded-SUM query R2 established, filtered by `created_at <= t` for historical standing — no snapshot job, no scheduler. A claim always targets the most recently completed month, never the in-progress one. Reward tiers, first pass (system design §21, corrected 2026-07-20): top 1 gets 500 gold + the existing curated Warhammer — the exclusive title is deferred, since `characters.equipped_title` doesn't exist until Epic S; top 2-10 get 100 gold + one Repair Token + one Pet Care Kit; top 11-100 get 30 gold; below top 100, no reward. Claim request re-validates standing server-side at claim time (never trusts a client-reported rank) and rejects if already claimed for the period, enforced via `ladder_claims` (`character_id, period_start, claimed_at`, `UNIQUE(character_id, period_start)`, its own V14 migration since V13 went to R1-R2) — same shape as `quest_claims`.
Priority: P2. Size: M. Status: done (2026-07-20). *(Depends on R2 for ranked Elo to exist. Reward items referencing pets/weapons/tokens depend on Epic O's catalog existing. Shipped as `V14__Add_Ladder_Claims.sql` + `LadderService`; the top-1 exclusive title stays deferred to Epic S.)*

## Epic S — Achievements & Character Page — new, eighth slice of the 2026-07-10 a–k scope expansion, redesigned 2026-07-15

Replaces the existing achievement scaffold rather than extending it — `achievement_definitions`/`account_achievements` already exist with 10 seeded achievements (this doc previously said 9; corrected 2026-07-20 while grounding S1-S2 against the actual seed migration), but nothing in the codebase ever writes an unlock; every achievement is permanently locked today. See system design §22 for the full reasoning, including the account-vs-character scoping question and Steam forward-compatibility notes.

Split into an S1+S2 implementation prompt (content model + automatic unlock) and a deferred S3+S4 prompt (character page + title equip), the same way R1-R2/R3 were split — S3/S4 are read-mostly aggregation that only makes sense once unlocks actually exist. S1+S2 written as `docs/planning/18-implementation-prompt-s1-s2-achievements.md` (2026-07-20), merged 2026-07-21 (PR #14). S3+S4 written as `docs/planning/19-implementation-prompt-s3-s4-character-page.md` (2026-07-21).

**S1.** As a designer, achievement content is defined declaratively — scope, criteria, and rewards as data, not hand-written per-achievement code.
Acceptance criteria: `achievement_definitions` gains `scope` (`ACCOUNT`/`CHARACTER`), `criteria_type` + `criteria_params JSONB` (v1.0 vocabulary: `TOTAL_WINS`, `WIN_STREAK`, `FASTEST_WIN_TURNS`, `CHARACTER_LEVEL`, `ITEM_ACQUIRED`, `ACCOUNT_CREATED_BEFORE`), `category`, `hidden`, `points`, and generalized rewards (`gold_reward`, `badge_id`, `title_id`, `bonus_character_slots`, `bonus_daily_battles`, alongside the existing `xp_reward`). `account_achievements` renamed to `achievement_unlocks`, `progress_data` dropped, nullable `character_id` added, uniqueness enforced via two partial indexes (one per scope) rather than a single constraint. Also adds `battle_history.turn_count` here rather than in S3 — S2 itself needs it for `FASTEST_WIN_TURNS`. Migration `V15__Redesign_Achievements_And_Character_Statistics.sql` (V14 went to R3's `ladder_claims`). The 10 existing seeded achievements get real `scope`/`criteria_type`/`criteria_params`/`points` values assigned as part of this story's content work.
Priority: P0. Size: M. Status: **done** (2026-07-20). Shipped as `V15__Redesign_Achievements_And_Character_Statistics.sql` + the `server.achievement` package (enums, facts records, `AchievementEvaluator`) and `AchievementDefinitionEntity`. All 10 seeded achievements carry real `scope`/`criteria_type`/`criteria_params`/`points`. *(One deviation from the spec: the two unlock inserts use a bare `INSERT ... ON CONFLICT DO NOTHING` — Postgres honors the partial indexes without an explicit target, and it's the one form that also runs against the H2 test DB, which supports neither partial indexes nor `ON CONFLICT (cols) WHERE`.)*

**S2.** As a player, achievements unlock automatically when I meet their criteria, and I receive their rewards immediately.
Acceptance criteria: a pure `AchievementEvaluator` (Ashley/DB-free, same shape as `ActionResolver`) interprets `criteria_type`/`criteria_params` against live-computed state — no stored progress anywhere. In this codebase leveling and the item-grant hook only ever happen as a consequence of a battle, so the design doc's four checkpoints collapse to two real call sites: one evaluation call at the end of `RewardService`'s existing transaction covering `TOTAL_WINS`/`WIN_STREAK`/`FASTEST_WIN_TURNS`/`CHARACTER_LEVEL`/`ITEM_ACQUIRED` together, and a separate one at account creation for `ACCOUNT_CREATED_BEFORE`. Unlock write is `INSERT ... ON CONFLICT DO NOTHING` against the appropriate partial index, atomic with reward application — idempotent, and the conflict clause itself is the only "already unlocked" check (no separate read-then-write). XP/gold rewards target the triggering character; `bonus_character_slots`/`bonus_daily_battles` reuse Epic Q's grant paths, their first real caller.
Priority: P0. Size: L. Status: **done** (2026-07-20). Stateless `AchievementUnlockService` (rides the caller's own `Handle`) evaluated at the end of `RewardService`'s transaction (character scope) and in its own small transaction at account creation in `UserService` (account scope, never blocking login on failure). `bonus_character_slots`/`bonus_daily_battles` are now Epic Q's first real caller. No client packet surfaces unlocks yet — that's S3. *(Depends on S1. Also depends on L1 for level-up detection and L3 for the item-grant hook.)*

**S3.** As a player, I can view a character page showing my stats, battle history, and unlocked achievements.
Acceptance criteria: `battle_history.turn_count` (added in S1's V15 migration, not a separate migration here — S2 needs it too) is populated from the turn count `AttackService` already computes but never persisted. Character page is a new read-only aggregate request: character info + level/exp (Epic L) + live-computed statistics (total wins/losses, win percentage, current win streak, fastest win by `turn_count`, last 5 matches) + unlocked achievements/badges (account-scoped ones plus this character's own character-scoped ones) + Achievement Score (sum of unlocked `points`) + equipped title. No new persistence beyond `turn_count` — everything else is computed live from existing tables.
Priority: P1. Size: M. Status: **done** (2026-07-21). `CharacterPageService` (takes the raw `Jdbi`, LadderService-style) assembles the read-only aggregate in one request: live statistics (total wins/losses, win %, current streak via reused `RewardService.currentStreak`, fastest win by `turn_count`, last 5 matches) + every achievement with unlock status + Achievement Score + equipped title. Bot opponents in match history surface a fixed `"Unknown Challenger"` placeholder (applied in the service, not the DAO) so they stay indistinguishable from real opponents. Server-side only; no client packet wiring yet (M4). *(Depends on S1/S2 for achievements to have unlockable state.)*

**S4.** As a player, I can equip a title I've unlocked so it displays on my character page.
Acceptance criteria: `characters.equipped_title VARCHAR(50)` (V16, S3/S4's implementation prompt); a set-title request validates the requested title is unlocked for the character's account (join against `achievement_unlocks`/`achievement_definitions`) before allowing it — honored for any character on the account when the granting achievement was ACCOUNT-scope, not just whichever character triggered the unlock. Ownership-guarded.
Priority: P2. Size: S. Status: **done** (2026-07-21). `V16__Add_Equipped_Title.sql` adds `characters.equipped_title`. `CharacterPageService.setEquippedTitle` validates the title is unlocked for the account/character via the OR-by-scope join against `achievement_unlocks`/`achievement_definitions` (an ACCOUNT-scope title is wearable by any character on the account; `TITLE_NOT_UNLOCKED` otherwise), clearing (`titleId == null`) always succeeds. Ownership-guarded, server-side only. This closes out Epic S. *(Depends on S1/S2 for title-granting achievements to exist.)*
